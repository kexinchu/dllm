# NIPS 2027 论文规划笔记

## 0. 工作标题与一句话定位

### 暂定标题
- `Precision Only Where It Matters: Selective High-Precision Reductions for Reproducible LLM Inference`
- 中文可表述为：`只在关键规约处引入高精度：面向可复现 LLM 推理的选择性高精度规约方法`

### 一句话定位
- `LayerCast` 的核心思路是将推理计算整体提升到 `FP32`，以降低数值误差导致的输出分叉。
- 我们的核心观察是：端到端 non-deterministic amplification 的主导来源并不是所有 compute path，而是少量关键 `reduction / accumulation path`。
- 因此，没有必要让所有算子都在高精度下执行；只要在关键规约位置同时做到 (a) FP32 累加器 + (b) batch-invariant 的固定 reduction plan，就可以在保持更高效率的同时，获得接近 `LayerCast` 的 reproducibility。

### ⚠️ 2026-04-25 概念修正：源码层面的事实校对

经源码核对（HF transformers / PyTorch ATen / Dao-AILab FlashAttention-2）：

| 现代默认行为 | 源码证据 |
|---|---|
| HF `LlamaRMSNorm` 已经 **FP32 累加器** 做 variance reduction | [`modeling_llama.py:62-67`](https://github.com/huggingface/transformers/blob/main/src/transformers/models/llama/modeling_llama.py)：`hidden_states.to(torch.float32).pow(2).mean(-1, keepdim=True)` |
| PyTorch SDPA 任何后端（FA-2 / EFFICIENT / CUDNN / MATH）都用 **FP32 累加器** | FA-2 [`kernel_traits.h:17`](https://github.com/Dao-AILab/flash-attention/blob/main/csrc/flash_attn/src/kernel_traits.h)：`using ElementAccum = float;`（硬编码）|
| cuBLAS BF16 GEMM **永远** 用 FP32 inner-product 累加器（`CUBLAS_COMPUTE_32F` 始终开）| PyTorch `aten/src/ATen/cuda/CUDABlas.cpp:625-635` |
| `torch.backends.cuda.matmul.allow_bf16_reduced_precision_reduction=False` 实际控制的是 **split-K 跨段聚合**的精度（不是内积累加器）| 同上：操作的是 `CUBLASLT_MATMUL_PREF_REDUCTION_SCHEME_MASK`，可选 `OUTPUT_TYPE`（BF16）/`COMPUTE_TYPE`（FP32）/`NONE`（禁 split-K） |

**因此**：FP32 累加器对绝大多数 site 是默认状态，没什么可"提升"的。**真正决定 batch-invariance 的不是累加器精度，而是 reduction plan 是否随 bs 改变**：

| Site | FP32 累加器? | Batch-invariant plan? |
|---|---|---|
| MatMul-K | ✅ 默认 | ❌ cuBLAS heuristic 按 bs 选 split-K |
| Norm-Stat (RMSNorm) | ✅ 默认 | ✅ per-row 无跨 row reduction |
| Softmax-Red (attn) | ✅ 默认 | ❌ FlashDecoding 在长 KV 时按 bs 选 split-KV |
| Softmax-Red (非 attn) | ✅ 默认 | ✅ per-row |

> `legacy/diagnose_fp32_matmul_bs_invariance.py` 实测：FP32 cuBLAS matmul 在 ~18% 形状上 bs-variant —— FP32 累加器 alone 不够。

**Implication for SRP**：真正需要 patch 的只有两个 site：MatMul-K 和 Attention。Norm 和 non-attn Softmax patch 在 dense 模型上是冗余的（只对 vLLM / SGLang 等可能换 kernel 的 serving stack 有意义）。

## 1. 对目标论文的理解

### 1.1 论文主结论
- 参考论文：[Give Me FP32 or Give Me Death? Challenges and Solutions for Reproducible Reasoning](https://arxiv.org/pdf/2506.09501)
- 该论文指出：即便在 greedy decoding 下，LLM 推理结果仍会因为 `batch size`、`GPU 数量`、`GPU 类型` 等变化而发生明显分叉，尤其在 reasoning model 上更严重。
- 根因是浮点数运算的有限精度和非结合律，在多种 kernel / batch / hardware 配置下，规约顺序变化会引入微小误差；这些误差在自回归推理中会被逐步放大。
- 论文提出 `LayerCast`：参数仍可用低精度存储，但推理中的计算路径整体提升到 `FP32`，从而显著提高 reproducibility。

### 1.2 这篇论文给我们的启发
- 它已经证明了一个重要事实：`更高的计算精度` 确实可以显著缓解 LLM inference 的 non-determinism。
- 但它并没有充分回答另一个更细的问题：
  `是否真的需要将所有计算都提升到 FP32，还是只需要修复少量关键数值敏感点？`
- 这正是我们工作的切入点。

### 1.3 我们的区别
- 我们不再把问题建模为一个全局开关：`BF16` vs `FP32-all`
- 我们把问题建模为：`precision allocation on sensitive reductions`
- 也就是说，我们的目标不是证明“FP32 很好”，而是证明：
  `只在关键 reduction 上提升精度，就足以恢复大部分 reproducibility；全局 FP32 是一种代价过高的过度修复。`

## 2. 论文核心假设与贡献表述

### 2.1 建议的核心假设（2026-04-25 修订）
- 浮点 non-determinism 的源头来自非结合律 + reduction order 改变；reduction order 改变在真实 LLM 推理栈里又来自两件事：
  1. **kernel 选择随 bs 改变**（cuBLAS split-K / FlashDecoding split-KV 的 heuristic 按 M / KV 长度选不同切分）
  2. 累加器精度不够（即使 plan 固定，BF16 累加器也会随小数位差异翻转最终 BF16 round 结果）
- 现代 PyTorch + HF + FA-2 已经把 **#2 解决了**（FP32 累加器是默认）。**剩下的 batch-non-determinism 完全由 #1 主导**。
- 但在真实 LLM 推理栈中，并不是所有 reduction 都会受 #1 影响：只有 GEMM 和 attention 这两类**含 cross-segment reduction（split-K / split-KV）**的 op 才会按 bs 改 plan。
- 因此，在这两个位点提供 **batch-invariant 的固定 plan 实现**（同时保持 FP32 累加器，避免任何精度回退），即可在较小额外开销下达到接近 `LayerCast` 的 reproducibility。

### 2.2 建议的贡献写法
- 我们系统性地核查了现代 LLM 推理栈在不同 reduction 位点的默认精度和 plan 行为，**通过源码证据**指出 batch-non-determinism 的真正来源是少量含 cross-segment reduction 的位点（GEMM 的 split-K + attention 的 split-KV）按 bs 改 kernel plan，而不是大众想象的"累加器精度不够"。
- 我们提出 `Selective Reduction Precision` 方法 —— 在这两个位点提供固定 plan、FP32 累加器的 kernel 实现；其它 reduction 位点保持框架默认。
- 在多个 reasoning / instruction 模型、多个 benchmark、不同 batch size / GPU 数量 / GPU 类型设置下，我们的方法在保持更高吞吐和更低代价的同时，达到接近 `LayerCast` 的 reproducibility。

### 2.3 不建议过度声称的点
- 不要写：`我们证明 non-determinism 只来自 reduction`
- **不要写**：`只要把累加器升到 FP32 就足够`（事实上 FP32 已经是默认，问题在 plan）
- 更稳妥的说法是：
  - `floating-point non-associativity originates from reductions; in practice its dominant trigger in modern stacks is bs-dependent kernel-plan selection (split-K / split-KV)`
  - `our experiments show that only a small subset of reductions dominates end-to-end output divergence`
  - `selective fixed-plan reductions on these sites are sufficient in practice`

## 3. 论文主线

### 3.1 最推荐的叙事顺序
1. LLM inference reproducibility 是真实且严重的问题，reasoning model 尤其明显。
2. `LayerCast` 说明了高精度可以缓解问题，但其代价较高。
3. 我们进一步追问：是否所有 compute path 都同等重要。
4. 我们发现，端到端输出分叉由少量关键 reduction 主导。
5. 基于该观察，我们提出只在关键 reduction 上提升精度的方法。
6. 该方法实现了更优的 `reproducibility-efficiency Pareto frontier`。

### 3.2 论文中真正需要证明的命题
- 不是：`reduction 会受非结合律影响`
- 而是：
  - `在真实 LLM 推理系统中，少量 reduction 位点主导了端到端输出分叉`
  - `只修复这些位点就足够`
  - `全局 FP32 并不是必要条件`

## 5. 方法定义与实验方法命名

### 5.1 建议统一方法名（2026-04-25 修订）
- `BF16`: baseline。HF 默认。**已使用 FP32 累加器**（HF RMSNorm / SDPA-FA-2 / cuBLAS COMPUTE_32F），但 GEMM 的 split-K 和 attention 的 split-KV 都按 bs 改 plan
- `FP32-all`: 所有相关计算尽可能使用 `FP32`（model 加载时 `dtype=torch.float32`）；既改累加器又改 storage
- `LayerCast`: 按原论文方法实现 —— BF16 weights, per-Linear FP32 cast → FP32 matmul → BF16 output
- `SRP-FP32`: 在关键 reduction 位点提供 **batch-invariant 的固定 plan kernel**，accumulator 仍为 FP32（与 BF16 baseline 一致）。**不是"提升累加器精度"，而是"固定 reduction plan"**
- `SRP-site-ablation`: 只对某一类 reduction 做 fixed-plan replacement
- `BF16-FP32-cross-split`: cuBLAS flag = False。控制变量：仅强制 BF16 GEMM 的 split-K 跨段聚合用 FP32（不动 plan 选择）。用作"flag-only"对照

注：原计划包含 `SRP-FP64`。**已于 2026-04-25 删除**。原因：A6000 / Ampere 消费卡的 tensor core 不支持 `BF16 input → FP64 accumulator` 路径；任何 FP64 实现必须把 input 整体 upcast 到 FP64（标量 ALU 计算）。这样 FP64 的开销由 "FP64 ALU 慢 ~60×" 主导，**而非 reduction precision 本身**——与 RQ4 想测的 "selective reduction precision FP32 vs FP64" 不再是干净对照。如果将来在 H100/A100（强 FP64 + tensor core 支持）上测，再恢复。

### 5.2 reduction 位点分类（2026-04-25 修订）
| Site | 是否真有 batch-dependent plan | 我们要不要 patch |
|---|---|---|
| `MatMul-K` GEMM 沿 K 维累加 | ✅ cuBLAS heuristic 按 M 选 split-K | **要**：固定 plan kernel（Triton no-split-K，或 cuBLASLt pinned algo）|
| `Softmax-Red` attention softmax (max/sum) | ✅ FlashDecoding 在长 KV 时按 (bs, KV_len) 选 split-KV | **要**：Triton FA-style 无 split-KV |
| `Attn-Value-Red` attention score@V | ✅ 同 Softmax-Red 在同一 fused kernel 里 | **要**：与 Softmax-Red 一起在同一个 Triton attention kernel 实现 |
| `Norm-Stat` RMSNorm/LayerNorm 统计量 | ❌ HF 默认 per-row data-parallel，无跨 row reduction | **不用**：HF 实现已 FP32 + batch-invariant |
| `Softmax-Red (非 attn)` LM head / MoE gate | ❌ per-row | **不用**：PyTorch 默认已 FP32 + batch-invariant |
| `CrossGPU-AllReduce` TP/EP 通信归约 | ✅ NCCL 算法（ring/tree）按 size / topology 选 | 单卡环境暂跳过；future work |

**真正需 patch 的只有 MatMul-K + Attention（两项）**——这是 SRP 在 dense LLM 上的"必要最小集"。

### 5.3 论文必须讲清楚的实现细节
- 我们改的不是"输入临时上调精度"，也不是"累加器精度"（因为现代框架默认就是 FP32）。**我们改的是 reduction plan 的稳定性**：
  - 强制不走 split-K（GEMM 用固定 tile，逐 K 块顺序累加）
  - 强制不走 split-KV（FlashAttention 内部 KV 切分大小固定为常量，不随 bs 改变）
- 是 `intra-kernel reduction`（写自己的 kernel 控制 plan），不是 `cross-device reduction`（多卡 TP 暂时不在范围内）
- 对所有 transformer layer 一视同仁（不针对某些 layer）
- prefill + decode 都改（decode 阶段 split-KV 影响最大；prefill 主要是 split-K）

### E6. 精度等级消融：FP32 vs FP64  ❌ **已删除（2026-04-25）**

原计划想验证 `SRP-FP32` 与 `SRP-FP64` 的收益差异。**因为 A6000/Ampere 消费卡 tensor core 不支持 BF16→FP64 累加器路径**，任何 FP64 实现的开销由 ALU 路径主导（A6000 FP64 throughput ≈ 1.16 TFLOPS，BF16 tensor core ≈ 78 TFLOPS，相差 60×+），与 reduction precision 无直接关系。

去掉这个实验后，**RQ4 简化为：`SRP-FP32` 是否已足够接近 `LayerCast` 的 reproducibility？** 论文中如对 FP64 提一句，可放在 Limitations / Future Work：在支持 BF16→FP64 tensor core 的硬件（如 H100 + 特殊 dtype 配置）上重做该 ablation。

## 11. 最重要的 reviewer 风险点

### 风险 1
- Reviewer 可能会说：
  `非结合律本来就主要出现在 reduction，为什么这也算贡献？`

### 应对
- 不要把贡献写成“发现 reduction 是根因”
- 真正贡献应写成：
  `在真实 LLM 推理系统中，只有少量关键 reduction 位点主导端到端输出分叉，我们识别了这些位点，并证明只修复这些位点就足够。`

### 风险 2
- Reviewer 可能会说：
  `你的方法只是一个 heuristic，为什么不是碰巧有效？`

### 应对
- 做 `site ablation`
- 做 `不同模型 + 不同数据集 + 不同硬件`
- 做 `decode-only / prefill-only / both`

### 风险 3
- Reviewer 可能会说：
  `你说接近 LayerCast，但是否在所有设置下都成立？`

### 应对
- 主文中至少放两类模型、两类数据集
- 附录补充更多设置
- 主文强调 `near-LayerCast reproducibility under practical settings`

### 风险 4
- Reviewer 可能会说：
  `你是否真的 deterministic？`

### 应对
- 不要轻易用 `deterministic inference`
- 更稳妥的表述：
  - `improved reproducibility`
  - `near-deterministic behavior`
  - `substantially reduced output divergence`

## 14. 当前建议的核心结论表述

- `LayerCast` 证明了全局高精度可以缓解 LLM inference 的 non-determinism，但代价较高。
- 我们的实验表明，端到端输出分叉主要由少量关键 reduction 位点主导，而非所有 compute path 共同决定。
- 基于这一观察，我们提出仅在关键 reduction 路径引入高精度规约的 `SRP` 方法。
- 该方法在多个模型、数据集和系统配置下实现了接近 `LayerCast` 的 reproducibility，同时显著降低了吞吐、延迟和显存方面的额外开销。

## 15. ⭐ 2026-04-26 头条实证结果：vs vLLM 官方 batch_invariant_ops

5-题 DeepSeek-R1-Distill-Qwen-7B / MATH500 / greedy 256 tokens / bs ∈ {1,2,4,8,16}：

| Method | Avg_Std@top1 | Total wall | vs HF-BF16 | vs vLLM-BF16 |
|---|---|---|---|---|
| HF BF16 (DynamicCache) | 3.97e-3 | 194s | 0% (ref) | — |
| vLLM BF16 (cudagraph + flash-attn) | 2.60e-3 | 179s | -8% | 0% (ref) |
| **HF + SRP-FP32 (我方)** | **0** strict | **279s** | **+44%** | **+56%** |
| **vLLM + `VLLM_BATCH_INVARIANT=1` (官方)** | 2.55e-5 | 402s | +108% | +125% |
| FP32-all | 3.59e-7 | ~470s | +140% | +163% |

### 双重优势（头条 contribution）

- **Determinism**：我方在 token 级 *strict bit-exact*（构造性保证 — `Avg_Std=0` 不依赖于"误差恰好不放大"），优于 vLLM-BI 的 2.55e-5（仍含概率性放大）
- **Wall-clock**：我方 279s vs vLLM-BI 402s — **快 31%**

### 重要含义（论文论点）

> Achieving batch invariance does **not** require the vLLM serving stack with cuda graphs and Triton attention. Our approach — fixed-plan Triton kernels (no split-K, no split-KV, fixed `BLOCK_K=64`) inserted into the standard HF eager pipeline via `nn.Linear.forward` / `RMSNorm.forward` monkey-patches — achieves strictly stronger determinism (Avg_Std = 0) at lower wall-clock cost than vLLM's official `VLLM_BATCH_INVARIANT=1` mode.

### 失败路径（写进 §10 Lessons）

下列 vLLM 报告 ~34% overhead 时所依赖的优化，在 HF eager 路径下都试过、不可达：

| 尝试 | 结果 |
|---|---|
| cuBLASLt fixed-algo backend | Std 仅减 50% (2.0e-3)，**离 FP32 floor 太远**；不是 strict |
| Hybrid M-based dispatch (M≤16 cuBLASLt, M>16 Triton) | 同上；decode kernel 也不严格 bit-exact |
| `torch.compile(mode='reduce-overhead')` | dynamo 把 Triton kernel 当 graph break，无效 |
| 手工 CUDA graph capture | HF Llama/Qwen2 attention 含 Python `.item()` 同步点，capture 出错 |
| HF StaticCache + eager attn | StaticCache 单用是 +12-61% regression（我方加 SRP 后总 +44% 已优于此） |

→ vLLM 报的 34% overhead 是**他们 cudagraph + 自家 attention kernel 的 stack-specific 数字**，不是 batch_invariant 算法本身的下界。我方在 HF eager 拿到 +44% (+strict) 已是当前最优。
